# 论文阅读方法

## 原则

- 只定义分析顺序，不预置论文内容。
- 所有判断必须来自当次材料。
- 不做逐篇摘要，不按年份机械排序。

## 分析链

```text
目的
  -> 主体
    -> 旧瓶颈
      -> 关键动作
        -> 机制链
          -> 证据链
            -> 技术债 / 边界
```

先回答：

1. 研究目的：这项工作要把什么瓶颈、成本或能力缺口，变成什么可用收益？
2. 阅读主体：本次真正要解释的是论文、机制、系统、模型家族、数据集、benchmark，还是一个技术瓶颈？
3. 旧瓶颈：它出现前为什么做不到？
4. 关键动作：它到底改了什么？
5. 证据边界：哪些来自论文、哪些来自 repo、哪些只是推断？

## 输出结构

```text
目的与核心结论
旧世界为什么不够
关键动作
机制链
证据链
技术债
边界
```

## 开头组织

生成文档式长文或 HTML 时，开头使用：

```text
研究目的
  -> 开篇总览图
    -> 核心结论
```

- 研究目的先回答“对象 / 瓶颈 / 收益”。
- 开篇总览图用 SVG 或可视化结构图压缩全文读法，帮助读者先获得大概内容。
- 总览图不是目录，不重复章节名；只放当前材料抽出的场景压力、旧瓶颈、关键动作、证据方向或结论落点。
- 核心结论再展开成完整判断，承接总览图进入正文。

## 公式处理

遇到核心公式时，不要只把公式塞进普通段落。按”教科书式解释”展开：

```text
公式回答什么问题
  -> 每个符号分别代表什么
    -> 分子 / 分母 / 约束条件的直觉
      -> 这个公式为什么导向论文的关键动作
```

- 先说明公式在算什么，不先堆符号。
- 逐项解释变量，尤其说明”越大越好 / 越小越好 / 只是中间成本”。
- 把公式连接回论文目的和机制链，避免变成孤立数学说明。
- 只解释当次材料里真正支撑结论的公式；次要公式可以保留为普通行内说明。

### LaTeX 源码要求

核心公式生成 `formula-card` 时，**必须同时提供 LaTeX 源码**，写入 `data-katex` 属性：

```text
论文公式 α = exp(z) / Σ exp(z)
  → data-katex=”\alpha = \frac{\exp(z)}{\sum \exp(z)}”
  → equation-plain: “α = exp(z) / Σ exp(z)”
```

- LaTeX 只用 KaTeX 支持的子集：`\frac`、`\sum`、`\int`、`\begin{pmatrix}`、`^{}``_{}`、`\alpha` 等。
- 不要用文档级命令（`\begin{equation}`、`\section`、`\label`）。
- `equation-plain` 纯文字版用 Unicode 近似（α、Σ、√、≤），必须手写，不能留空。
- KaTeX 渲染由前端自动完成，详见 `docs/04-katex-setup.md`。
- 从论文 PDF 或 repo 提取公式时，优先保留原始 LaTeX；只有原始格式不可用时才手写近似 LaTeX。

### 多公式场景

当论文涉及多个公式的推导链、变量依赖或损失组合时，单个 `formula-card` 不够。用 `svg-figure formula-relation` 画公式之间的关系：

```text
2 个以上公式的推导顺序    → 推导链（derivation-chain）
变量在公式间的流动关系    → 变量依赖图（variable-dependency）
总损失 = 各项之和          → 损失组合图（loss-composition）
两种方法的公式对比        → 方法对比图（formula-compare）
```

- 单个公式用 `formula-card` + KaTeX 渲染数学排版。
- 多个公式的关系用 `formula-relation` SVG 表达结构。
- 两者配合：`formula-relation` 的节点放公式缩写，相邻的 `formula-card` 放完整公式 + 变量解释。
- 模板和规则详见 `docs/04-katex-setup.md` 的"公式关系图"章节。

## 概念分段

概念底座要按“一个概念一段或一小节”组织，不把多个陌生词、机制约束和结论塞进同一段。

- 先定义概念，再讲它解决什么问题，再讲它和下一概念的关系。
- 如果同一段出现多个关键术语，优先拆成多个小标题或短段。
- 英文术语第一次作为重点出现时，附“英译中 / 怎么理解”，把字面翻译转成可操作含义。
- 抽象机制需要图解时，用小型 SVG 表达动作链或取舍关系，不只用文字解释。
- 划线解释只补读者缺口，不重复正文已有定义。
- 一个标记回答只解释当前标记，不顺手解释整条机制链。

## SVG 图解触发

稍微复杂的内容默认用 SVG 图解，不等用户明确要求“作图”。复杂度判断：

- 需要解释 3 个以上步骤、角色、模块或变量。
- 有“先后顺序、前置条件、失败链、取舍关系、预算分配、调度决策”。
- 同一段里出现多个技术对象并需要比较。
- 公式背后有分子 / 分母 / 约束 / 优化目标。
- 只用文字解释会超过 2 个自然段，或读者需要在脑中拼流程。

SVG 图解只画必要结构：

```text
输入 / 前提
  -> 关键动作
    -> 中间约束
      -> 输出 / 结果
```

- 图后必须写“这张图怎么读”。
- 图是正文解释的一部分，不是装饰。
- 图内文字短，不把长段解释塞进 SVG。
- 能用动作链表达时优先画动作链；能用对照表达时画左右对照；能用公式表达时画变量关系。

## 自检

- 开头没有研究目的，重写。
- 文档式 HTML 开头没有总览图，且读者需要先建立全局理解，补一张开篇总览图。
- 核心公式没有独立解释变量和直觉，补成公式解释组件。
- 核心公式的 `formula-card` 缺少 `data-katex` LaTeX 源码或 `equation-plain` 纯文字回退，补全双层渲染。
- 概念底座把多个关键术语堆在同一段，拆段或拆小标题。
- 关键英文术语没有“英译中 / 怎么理解”，补上。
- 抽象概念仍然只靠长文字说明，补 SVG 动作图或关系图。
- 稍微复杂的机制、对照、公式或调度逻辑没有 SVG 图解，补图。
- 研究目的只有“提升性能/提高效率”，重写到“对象 / 瓶颈 / 收益”。
- 没有旧瓶颈到关键动作的因果链，重写。
- 把资料包、年份或论文名气当主体，重写。
- 可见标题出现内部工作词，改成读者能懂的标题。
